# S0-S6 七阶段蒸馏流程

> 每个阶段有明确产出与检查点。建议顺序执行，但 **S2–S5 可迭代往返**。
> 发现缺口 → 回到 S2–S5 迭代修复 → 重新验证。

```
S0 盘点 → S1 对象 → S2 规则 → S3 API → S4 明细 → S5 编排 → S6 验证
                          ▲__________________________________|
                                 （发现缺口则迭代）
```

---

## S0. 准备与盘点
**做什么**：填写资料盘点表，确定事实优先级。通读原始需求，建立对系统"是什么"的整体印象。

**产出**：
- 资料盘点表（6 类资料：需求/Schema/实例数据/本体/服务层源码/路由层源码，标注是否具备+位置+可信度+用途）
- 一段话的系统定位

**检查点**：
- [ ] 资料盘点表已填
- [ ] 事实优先级已定（实际行为：源码＞DB＞本体＞需求；业务意图：需求＞本体＞源码注释）
- [ ] 资料不全的降级策略已确认（见 `fact-priority.md`）

---

## S1. 识别业务对象与边界（语义骨架）
**做什么**：从需求 + 本体 + 数据库表，归纳核心业务对象（聚合根、明细、主数据）。厘清对象间关系（1:N、N:N、引用 vs 组合）、生命周期/状态机。划清系统边界：管什么、不管什么。

**产出**：
- `00-overview.md` 初稿
- `domain/_index.md`
- 各 `domain/<对象>.md` 初稿

**检查点**：
- [ ] 每个对象能用一句话定义
- [ ] 关系图自洽
- [ ] 外部集成点（OAuth/webhook/第三方回调）已识别并单独建 domain 文档（不与普通 CRUD 混）

---

## S2. 抽取业务规则并标注执行方
**做什么**：从源代码服务层抽取实际强制的规则（校验、状态流转、自动计算）。与本体声明的约束比对，发现"声明了但代码没强制"的，记入冲突表。给每条规则打【服务端强制】/【服务端维护】/【调用方需预判】/【副作用】标签。

**产出**：
- 各对象文档的"核心业务规则"段
- `reference/model-code-conflicts.md`

**检查点**：
- [ ] 每条规则都有执行方标注
- [ ] 冲突已登记
- [ ] 配置类/集成点已读（`*Properties.java`/`*.yml`/`*.properties`），每个配置项追溯到用途

---

## S3. 定义能力层 API（用例级）
**做什么**：从"需求中的业务动作"出发列出用例（录入、开票、收款、查询、统计…）。对照路由层/服务层，确认每个用例对应的真实接口（方法+路径+入参+出参）。用例级粒度：必要时把多个原子接口合并描述为一个业务动作；剥离 UI 态字段。

**产出**：
- `api/_index.md`
- 各 `api/<域>/<用例>.md`

**检查点**：
- [ ] 每个用例能回答"什么场景用、输入什么、输出什么、有哪些规则、可能哪些错误"
- [ ] 接口粒度按业务动作，不按 CRUD
- [ ] 字段/参数与代码实际入参出参逐字段核对（不可臆测）

---

## S4. 整理明细层（Schema / 枚举 / 错误码）
**做什么**：字段级 Schema，逐字段标"对应哪个领域属性/API 字段"。汇总所有枚举/字典值，标注 ✅（代码在用）/ ️（仅模型定义、未实现）。汇总错误码与统一响应信封。

**产出**：
- `reference/schema/*.md`
- `reference/enums.md`
- `reference/error-codes.md`

**检查点**：
- [ ] API 文档里出现的每个枚举/错误码，明细层都能查到
- [ ] Schema 最后一列"对应领域属性/API 字段"已填（把 DB 命名和 API 命名对齐）

---

## S5. 建立交叉引用与编排说明
**做什么**：用 `[[slug]]` 把对象↔接口↔Schema 串起来。在 `api/_index.md` 写"自然语言意图 → 接口序列"的典型编排。标注每个写接口的"写权限分级"。

**产出**：
- 完善后的 `api/_index.md`
- `README.md`
- `requirements/_index.md`
- `capabilities.json`
- `index.html`（拷贝自 skill 的 `assets/viewer.html`——蒸馏文档查看器+领域对话客户端）
- `AGENTS.md`（AI 工具快捷入口）

**检查点**：
- [ ] 能顺着链接从"用户想做的事"走到"该调哪些接口、依赖哪些对象"
- [ ] 每个写接口有写权限分级（read-only / controlled-write / disabled）
- [ ] `capabilities.json` 工具条目与 API 文档逐字段对齐

---

## S6. 验证（实跑）
**做什么**：准备测试集，让 AI 仅凭 `distilled/` 推导调用，再真打到接口验证。修复发现的文档缺口，回到 S2–S5 迭代。

**产出**：验证记录 / 结论。

**检查点**：
- [ ] 静态清单全部通过（见 `verification.md` 8.1）
- [ ] 动态验证四类场景通过：读/查询、写/录入、统计、规则反例（见 `verification.md` 8.2）

---

## 迭代回路
S6 或 S2–S5 任何阶段发现缺口 → 回到对应阶段修复：
- 规则缺/错 → 回 S2
- API 缺/错 → 回 S3
- 明细缺/错 → 回 S4
- 编排/引用断链 → 回 S5

修复后重新跑 S6 验证，直到所有检查点通过。
