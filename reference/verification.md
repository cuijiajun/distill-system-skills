# 质量验收：如何确认蒸馏对了

## 8.1 静态检查清单（产出后必过）
- [ ] 每个领域对象有：一句话定义、状态机、规则段
- [ ] 每条规则有执行方标注（【服务端强制】/【服务端维护】/【调用方需预判】/【副作用】）
- [ ] 每个业务用例接口四要素齐全：场景 / 输入输出 / 规则 / 错误
- [ ] API 文档出现的每个枚举、错误码，明细层（`enums.md` / `error-codes.md`）都能查到
- [ ] 交叉引用无明显断链（`[[slug]]` 目标存在或标"待补"）
- [ ] 冲突清单 `model-code-conflicts.md` 已建立
- [ ] `capabilities.json` 参数与代码实际入参逐字段对齐
- [ ] 写权限分级已标注每个写接口
- [ ] **外部集成覆盖**：`@ConfigurationProperties`/`*.yml`/`*.properties` 全部已读；每个第三方回调/OAuth/webhook/上游地址 都有独立 domain 文档并标注外部依赖；每个配置项都追溯到用途
- [ ] **密钥脱敏**：distilled/ 全文 grep 无真实密码/token/secret/私钥（只有字段名 + `<见配置文件>` 占位）；构建产物（target/.class/jar/logs）未被读入
- [ ] **查看器已拷贝**：`assets/viewer.html` → `distilled/index.html`（缺失=用户无法预览，最易漏的最后一步）
- [ ] **项目根 AGENTS.md 已更新**：蒸馏完成后，在**项目根目录**的 AGENTS.md（无则新建）写入指向 `distilled/` 的引用段落，确保 opencode/codex/claude code 能发现蒸馏产物

## 8.2 动态验证（实跑）——强烈推荐
准备一组**真实场景测试集**，让 AI **仅凭 `distilled/`** 推导出应调用的接口序列，再真打到运行中的系统，核对结果。建议覆盖四类：

| 类型 | 示例 | 验证什么 |
|---|---|---|
| 读/查询 | "查 X 部门今年的合同" | 名称→ID 解析、查询参数 |
| 写/录入 | "给客户 A 录个合同……" | ID 解析、必填、用例编排、服务端维护字段 |
| 统计 | "未来三个月各部门预计收多少" | 统计接口口径理解 |
| 规则反例 | 故意违反规则（比例≠1、重复操作） | "关键规则/可能的错误"是否准确预测 |

**判定标准**：AI 能正确解析名称、推导前置条件、按正确顺序编排接口、预判并解释规则错误，且字段/枚举/错误消息与真实系统一致 → **文档可用性通过**。

## 迭代回路
发现文档缺口 → 回到对应阶段（规则 S2 / API S3 / 明细 S4 / 编排 S5）修复 → 重新实跑，直到典型读、写、统计、规则反例场景全部通过。
